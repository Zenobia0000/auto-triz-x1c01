# TRIZ Step 4: 原型驗證 + 複雜度檢查

## Overview

本 skill 實作 Auto-TRIZ 閉環流程的 Step 4（Section 6 of `docs/auto_triz_strategy.md`）。

接收 Step 2-3 產出的解法方案（F/S/OZ/OT + 分離策略 + 標準解），執行：
1. Px 分離驗證
2. 四問複雜度判定（補丁 vs 進化）
3. 新 TC 偵測
4. 下游交付物產出

**宣告：** 「正在使用 triz-verify skill — 執行 Step 4: 驗證與複雜度檢查。」

## Phase 1: Px 分離驗證

### 1.1 邏輯驗證

用 PC 造句檢查解法是否真正分離了 Px：

```
原始 PC：Px 必須是 [A]（為了 P1）且 [非 A]（為了 P2）
解法後：
  - 在 [分離維度的一側]，Px = A → P1 滿足 ✓/✗
  - 在 [分離維度的另一側]，Px = 非 A → P2 滿足 ✓/✗
```

### 1.2 驗證結果判定

| 結果 | 動作 |
|:-----|:-----|
| 兩側皆滿足 | 通過，進入 Phase 2 |
| 一側不滿足 | 分離不完整 — 回到 Step 3 調整分離策略 |
| 兩側皆不滿足 | 解法無效 — 回到 Step 2 重新選擇原理 |

### 1.3 實證驗證（Empirical Validation — 強制）

對工程規格書中的每個參數值，透過外部資料驗證其合理性：

```
對每個工程規格參數（F 數值、S 物性、延遲預測、效率目標等）：

1. WebSearch: "{技術/材料} {指標} benchmark OR published results OR state-of-art"

2. 根據搜尋結果分類：
   - [PLAUSIBLE]：聲明值在已發表基線範圍內
     → 記錄基線來源，Confidence = HIGH
   - [NOVEL_CLAIM]：無已發表基線存在
     → 標記「需原型驗證」，Confidence = LOW
   - [EXTRAORDINARY]：聲明值超越已發表 SOTA > 50%
     → 標記「需非凡證據」，Confidence = LOW
     → 建議具體的驗證實驗

3. 匯入 Evidence Registry，更新 Claim IDs
```

**判定影響：**
- Evolution 判定**要求** Evidence Registry 中 HIGH+MEDIUM 占比 ≥ 50%
- 若 LOW 占比 > 50%，即使四問判定為 Evolution，也降級為「Weak Evolution — 需實驗補強」
- Patch 判定可接受較低覆蓋率，但須在技術債中記錄未驗證項目

## Phase 2: 四問複雜度判定（量化 CCI 評分）

依序對四個維度評分（5 級量表：0.00 / 0.25 / 0.50 / 0.75 / 1.00）：

### Q1: 結構複雜度 (Structural)

| 分數 | 條件 |
|:-----|:-----|
| 0.00 | 淨組件數減少（移除了零件） |
| 0.25 | 組件數不變（現有零件獲得新功能） |
| 0.50 | 新增 1-2 個零件，但無新跨子系統介面 |
| 0.75 | 新增 1-2 個零件且有新跨子系統介面 |
| 1.00 | 新增 3+ 個零件或全新子系統 |

### Q2: 能量複雜度 (Energy)

| 分數 | 條件 |
|:-----|:-----|
| 0.00 | 能耗降低或轉用環境免費能量 |
| 0.25 | 能耗不變 |
| 0.50 | 能耗增加 < 10% |
| 0.75 | 能耗增加 10-50% |
| 1.00 | 需要新能量源/類型（如：新增冷卻泵、新電源） |

### Q3: 認知複雜度 (Cognitive)

| 分數 | 條件 |
|:-----|:-----|
| 0.00 | 依賴鏈縮短（輸入到輸出步驟更少） |
| 0.25 | 依賴鏈不變 |
| 0.50 | 新增 1 個條件分支或校準步驟 |
| 0.75 | 新增 2-3 個條件分支或需專業知識才能理解 |
| 1.00 | 需要全新心智模型或大量文件說明 |

### Q4: 演化對齊 (Evolution Alignment)

計算滿足以下三條演化趨勢的數量：
- 理想度增加（更多功能，更少成本/危害）
- 朝微觀發展（材料層級解決）
- 動態化增加（感測器驅動的狀態切換）

| 分數 | 條件 |
|:-----|:-----|
| 0.00 | 3 條趨勢全部滿足 |
| 0.33 | 滿足 2 條 |
| 0.67 | 滿足 1 條 |
| 1.00 | 0 條滿足 |

### 複合複雜度指數 (CCI)

```
CCI = 0.30 × Q1 + 0.25 × Q2 + 0.20 × Q3 + 0.25 × Q4
```

### 判定規則

| CCI 範圍 | 判定 | 動作 |
|:---------|:-----|:-----|
| 0.00 - 0.30 | **Strong Evolution** | 高信心進入交付物產出 |
| 0.31 - 0.55 | **Weak Evolution** | 進入交付物但標記高分項為未來改進方向 |
| 0.56 - 0.75 | **Conscious Patch** | 建議回 Step 3 重新深挖；若時間不允許，記錄技術債 |
| 0.76 - 1.00 | **Hard Patch** | 強烈建議返回 Step 3，解法可能使系統整體惡化 |

## Phase 3: 補丁 vs 進化路徑

### 進化路徑
解法品質良好，產出下游交付物（Phase 4）。

### 補丁路徑

```
判定為補丁
│
├─ 時間/成本允許重新深挖？
│  ├─ 是 → 建議回到 Step 3 (PC) 重新深挖
│  │       詢問使用者是否要返回
│  │
│  └─ 否（出貨壓力）→ 有意識地選擇補丁：
│     1. 記錄為「已知技術債」
│     2. 標記哪個 PC 未被真正解決
│     3. 排入下一代 TRIZ 待辦清單
│     4. 繼續產出交付物
```

---

## Phase 4: 新 TC 偵測

檢查解法是否引入新的技術矛盾：

```
新方案是否引入新的 TC？
│
├─ 否 → 完成
│
├─ 新 TC 與原 TC 形成迴圈
│  （解 A → 惡化 B → 解 B → 惡化 A）
│  └─ 進入迴圈診斷（檢查 OZ/OT/Px 重疊）
│     └─ 重疊 → 其實是同一個未正確定義的 PC
│     └─ 不重疊 → 兩個獨立 TC
│
└─ 獨立新 TC → 回 Step 1（螺旋上升）
```

## Phase 5: 下游交付物

### 5.1 工程規格書

將 F/S/OZ/OT 轉為可交付的工程規格：

```markdown
## 工程規格書

### 解法摘要
- **問題**: [原始 TC 描述]
- **解法**: [一句話描述]
- **分離策略**: [類型 + 邊界條件]

### 規格參數
| 參數 | 值 | 單位 | 來源 |
|:-----|:---|:-----|:-----|
| F (場) | [數值] | [單位] | [控制方程] |
| S (物質) | [材料] | [物性] | [資料來源] |
| OZ | [位置] | - | [設計圖] |
| OT | [時機] | [頻率/週期] | [操作條件] |

### Hard Constraints
- [ ] 不違反 [具體物理定律]
- [ ] 相容於現有系統 [具體介面]
```

### 5.2 驗證計畫

```markdown
## 驗證計畫

| 驗證項目 | 量測指標 | 判定標準 | 方法 |
|:---------|:---------|:---------|:-----|
| Px 分離有效 | [指標] | [閾值] | [實驗方法] |
| P1 改善 | [指標] | [目標值] | [量測方法] |
| P2 不惡化 | [指標] | [上限值] | [量測方法] |
```

### 5.3 Evidence Registry

彙整 Step 2-3 和 Step 4 中所有數值聲明的來源追蹤：

```markdown
## Evidence Registry

| Claim ID | Claim | Value | Source Type | Source | Confidence | Accessed |
|:---------|:------|:------|:------------|:-------|:-----------|:---------|
| C-001 | [描述] | [數值+單位] | Paper/Datasheet/Handbook/LLM_estimate | [引用] | HIGH/MEDIUM/LOW | [日期] |
```

**Confidence 定義：**
- **HIGH**：同儕審查論文、官方 datasheet、NIST/標準手冊
- **MEDIUM**：廠商技術文章、研討會論文、Wikipedia（有引用來源）
- **LOW**：LLM 參數記憶，未經外部確認

**強制規則：** Evolution 判定的工程規格書中，所有 HIGH/MEDIUM 占比必須 ≥ 50%。若 LOW 占比 > 50%，降級為 Weak Evolution 並標記需要實驗驗證。

### 5.4 技術債登記（補丁時必填）

```markdown
## 技術債記錄

- **日期**: [YYYY-MM-DD]
- **未解決的 PC**: [Px 必須是 A 且非 A]
- **當前補丁**: [描述]
- **補丁風險**: [在什麼條件下會失效]
- **建議的真正解法方向**: [基於分離原理的建議]
- **預估解決時間**: [下一代/下下一代]
```
